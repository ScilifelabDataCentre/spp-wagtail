# Portal Data page

This document describes the implementation used to expose dataset listings in the Swedish Pathogens Portal (SPP). It covers the data source, parsing layer, Wagtail page model, routing, templates, and download handling.

The Portal Data page is implemented as a Wagtail CMS page that surfaces dataset metadata from external repositories â€” currently MetaboLights â€” alongside a file browser and download facility backed by a Kubernetes Persistent Volume Claim (PVC).

---

## Background and rationale

The Swedish Pathogens Portal (SPP) provides a national entry point for discovering research outputs related to infectious disease and pandemic preparedness generated by Swedish research organisations. A core goal of the portal is to make datasets *discoverable*, not to act as the primary host for research data.

Data-producing units within SciLifeLab deposit files in authoritative external repositories such as MetaboLights or ENA. The portal's role is discovery and user experience: mapping repository records into a consistent interface and directing users to repository-hosted files for download. This is an intentional design decision documented in [ADR-0007](../architecture/decisions/0007-data-hosting-architecture.md).

The current implementation is a working proof of concept for this model. It reads study metadata from ISA-Tab investigation files made available through a PVC mount and surfaces them through a searchable, faceted listing. The data shown so far has been added manually as part of prototype development in collaboration with IDS.

The near-term roadmap, also described in ADR-0007, extends this model to support unit-produced datasets that are not yet available in an external repository. The planned approach uses a precompiled S3 bucket with a lightweight index. That work will introduce a scheduled ingestion job, manifest validation, and pre-signed S3 download URLs. The Portal Data page UX is designed to accommodate both the current file-based approach and that future S3-backed model.

Long term, the plan is for SciLifeLab units to be able to deposit newly generated pathogen-related data directly into the portal.

## Architecture overview

The implementation is split across two locations that have distinct responsibilities:

```
portal_data/          Python package â€” data access and business logic
  services.py         Parses investigation files, searches, builds facets
  context.py          Builds the template context dict from a request
  views.py            View functions for the file browser and download

cms/pages/
  portal_data.py      Wagtail page model â€” routing, CMS fields, rendering
```

`portal_data` is implemented as a plain Python package rather than a Django app. It has no models, migrations, or URL configuration of its own. Django-facing concerns, including the page model, URL routing, and templates, are handled by the `cms` app.

## Data source

Studies are read from the filesystem at the path configured by the `DATASETS_ROOT` environment variable. Each study is expected to follow the ISA-Tab layout used by MetaboLights:

```
$DATASETS_ROOT/
  MTBLS1464/
    i_Investigation.txt
    s_MTBLS1464.txt
    a_MTBLS1464_LC-MS_positive_...txt
    FILES/
      ...
```

`services.py` locates and parses `i_Investigation.txt` for each study directory it finds. In practice, this file contains the study-level metadata used for the listing page. It prefers the latest revision under `METADATA_REVISIONS/` if that directory exists, falling back to the top-level file.

### HTML sanitisation

Study descriptions in the investigation file contain HTML authored by depositors. This is sanitised at parse time using `nh3` (a Python binding to the Rust `ammonia` library) before it reaches a template. This allows depositor-authored descriptions to be rendered without trusting arbitrary HTML. Only a safe presentational subset of tags is allowed: `p`, `br`, `strong`, `em`, `a`, `ul`, `ol`, `li`. Links keep their `href` and `target` attributes, but the `rel` attribute is stripped.

## Page model and URL routing

`PortalDataPage` in `cms/pages/portal_data.py` extends both `RoutablePageMixin` and `Page`. This means the page handles all its own sub-paths without any entries in the root URL configuration. Three routes are defined:

| Route | Method | Description |
|---|---|---|
| `""` | `index` | Dataset listing with search and facets |
| `<slug:accession>/files/` | `study_files` | File browser for one study |
| `<slug:accession>/files/<path:relpath>/` | `download_file` | Stream a single file |

Because this is a Wagtail page, it can be placed anywhere in the Wagtail page tree. Its URL is determined by the slug an editor gives it; no hardcoded prefix exists in `core/urls.py`.

The `index` route delegates context building to `build_portal_data_context` (in `portal_data/context.py`) and handles HTMX partial swaps itself. The `study_files` and `download_file` routes delegate entirely to standalone functions in `portal_data/views.py`, receiving the page instance or data type string as arguments so the view functions stay decoupled from the CMS.

### CMS fields

| Field | Purpose |
|---|---|
| `datatype` | Slug identifying the data type, e.g. `metabolomics`. Controls which `DataTypeConfig` is loaded from `SUPPORTED_TYPES` in `services.py`. |
| `default_page_size` | Number of datasets shown per page. |
| `content` | StreamField for editor-authored content (rich text, alerts) rendered above the search form. |

## Request lifecycle

A typical listing request flows through the stack as follows:

```
Browser â†’ Wagtail â†’ PortalDataPage.serve()
    â†’ PortalDataPage.index()
        â†’ PortalDataPage.get_context()
            â†’ build_portal_data_context()         # context.py
                â†’ get_dataset_listing()           # services.py
                    â†’ load_all_items()            # parses filesystem
                    â†’ search_items()              # filters by query
                    â†’ apply_facet_filters()       # filters by checkboxes
                    â†’ build_facets()              # computes facet counts
        â†’ render("cms/pages/portal_data/index.html")
```

For HTMX requests, such as checkbox changes, search input, or pagination, the same backend flow runs. The difference is that the response only renders `cms/pages/portal_data/partials/listing.html`, targeting the `#portal-data-listing` div in the browser.

## Templates

All templates live in `cms/templates/cms/pages/portal_data/`.

```
index.html              Full page template with the search form, HTMX attributes, and shared JavaScript
partials/
  listing.html          Partial returned during HTMX swaps; contains facets and result cards
study_files.html        File browser table for a single study
```

### HTMX and JavaScript

`index.html` includes a single delegated `click` listener on `document` that handles two interactions:

- **Facet "Show more / Show less"** â€” toggles the `hidden` class on the overflow list in each facet panel.
- **Description "Show more / Show less"** â€” toggles the `line-clamp-4` Tailwind class on the description div, expanding or collapsing it.

Using document-level delegation means both handlers continue to work after HTMX replaces `#portal-data-listing`, without needing to be re-attached.

### Facet labels

Facet field names, such as `design_types`, are converted to readable labels, such as `Design Types`, in `context.py` by `_build_facet_list`, which calls `_facet_label`. This happens server-side so no custom template tag is needed.

## Adding a new data type

Adding a new data type is mostly configuration, as long as the new type uses the same ISA-Tab investigation file format.

1. Add a new `DataTypeConfig` entry to `SUPPORTED_TYPES` in `services.py`, specifying a label and the facet fields to expose.
2. Add the new slug as a choice on the `datatype` field in `PortalDataPage`.
3. Create a new `PortalDataPage` instance in the Wagtail admin and set `datatype` to the new slug.

No URL configuration or template changes are needed unless the new data type uses a different investigation file format.

## Path traversal protection

Before serving a download, `serve_download_file` in `portal_data/views.py` resolves the requested path and verifies that it remains inside the study directory before opening any file. Absolute paths and paths that escape via `../` are rejected with a 404. This check uses `Path.is_relative_to()` on the resolved (symlink-expanded) paths.
